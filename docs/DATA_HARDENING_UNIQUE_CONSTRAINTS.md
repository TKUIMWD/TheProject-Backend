# Data Hardening: Unique Constraint Preflight

Date: 2026-05-26

This runbook records the duplicate-data checks required before any deferred unique MongoDB indexes are added. Do not add unique constraints until these checks have been run against staging or production data and every duplicate group has an explicit cleanup decision.

## Candidate Unique Keys

| Collection | Candidate key | Why it matters | Current state |
| --- | --- | --- | --- |
| `users` | `email` | Login, password reset, and registration identity lookup. | Non-unique `{ email: 1 }` index only. |
| `users` | `username` | Display/identity conflict checks during registration and profile updates. | Non-unique `{ username: 1 }` index only. |
| `compute_resource_plans` | `name` | Registration assigns the `standard` plan by name. | Non-unique `{ name: 1 }` index only. |
| `vms` | `{ pve_node, pve_vmid }` | PVE VM identity should be unique per node. | Non-unique compound index only. |
| `vm_tasks` | `task_id` | Task status lookup assumes a single task per generated ID. | Non-unique `{ task_id: 1 }` index only. |

## MongoDB Duplicate Checks

Run the checked-in preflight command against the target database:

```bash
npm run data:check-unique-duplicates
```

The command uses `DBUSER`, `DBPASSWORD`, `DBHOST`, `DBPORT`, and `DBNAME` from the current environment. It is read-only. It exits with code `1` when any duplicate group is found, and exits with code `0` only when every candidate unique key is clean.

The equivalent manual `mongosh` checks are listed below. The output should be an empty array before adding a unique index.

```javascript
db.users.aggregate([
  { $match: { email: { $type: "string", $ne: "" } } },
  { $group: { _id: { $toLower: "$email" }, ids: { $push: "$_id" }, count: { $sum: 1 } } },
  { $match: { count: { $gt: 1 } } },
  { $sort: { count: -1 } }
]);

db.users.aggregate([
  { $match: { username: { $type: "string", $ne: "" } } },
  { $group: { _id: { $toLower: "$username" }, ids: { $push: "$_id" }, count: { $sum: 1 } } },
  { $match: { count: { $gt: 1 } } },
  { $sort: { count: -1 } }
]);

db.compute_resource_plans.aggregate([
  { $match: { name: { $type: "string", $ne: "" } } },
  { $group: { _id: { $toLower: "$name" }, ids: { $push: "$_id" }, count: { $sum: 1 } } },
  { $match: { count: { $gt: 1 } } },
  { $sort: { count: -1 } }
]);

db.vms.aggregate([
  { $match: { pve_node: { $type: "string", $ne: "" }, pve_vmid: { $exists: true, $ne: "" } } },
  { $group: { _id: { pve_node: "$pve_node", pve_vmid: { $toString: "$pve_vmid" } }, ids: { $push: "$_id" }, count: { $sum: 1 } } },
  { $match: { count: { $gt: 1 } } },
  { $sort: { count: -1 } }
]);

db.vm_tasks.aggregate([
  { $match: { task_id: { $type: "string", $ne: "" } } },
  { $group: { _id: "$task_id", ids: { $push: "$_id" }, count: { $sum: 1 } } },
  { $match: { count: { $gt: 1 } } },
  { $sort: { count: -1 } }
]);
```

## Cleanup Notes

- `users.email`: preserve the verified account when duplicate emails exist. If multiple verified accounts share the same normalized email, require a manual owner decision before mutation.
- `users.username`: prefer keeping the oldest active account with the exact username. Rename or merge later duplicates only after confirming course, VM, template, review, and writeup ownership references.
- `compute_resource_plans.name`: keep exactly one `standard` plan before registration depends on it. Repoint users from duplicate plan IDs before deleting duplicates.
- `vms.{pve_node,pve_vmid}`: confirm the real VM identity in Proxmox before deleting or merging records. Never assume the newest MongoDB document is authoritative.
- `vm_tasks.task_id`: task IDs are generated by workflow code; duplicate groups should be audited against VM creation/deletion history before deletion.

## Migration Order

1. Run the duplicate checks on a recent staging copy.
2. Document every non-empty duplicate group and the chosen cleanup action.
3. Apply cleanup through a reviewed migration script or controlled admin operation.
4. Re-run the duplicate checks and archive the empty results.
5. Add unique indexes in a small schema-only change with rollback notes.
6. Run the full backend gate before shipping the schema change.
